# Aula 11 — Comunicação Científica e Defesa Técnica de Projetos de Software

## Princípio Fundamental
> "A banca avalia rigor científico, não apelo visual ou storytelling emocional."
> "Projeto sem validação é apenas demonstração."

Defesa técnica = **defesa científica**, não venda de produto.

## Erros Mais Comuns dos Alunos
| Erro | Por quê está errado |
|------|---------------------|
| Mostrar interface sem explicar | Exibir telas não prova que o sistema resolve o problema |
| Falar de ferramenta (React, Node, Flutter) | Ferramentas não são argumentos científicos |
| Não apresentar métricas | Sem números, não há como avaliar desempenho ou impacto |
| Não explicar resultados | Resultados sem interpretação não constituem pesquisa |

> "Interface não substitui evidência."

## Estrutura Obrigatória de Defesa Técnica
```
01 PROBLEMA     → O que se busca resolver
02 HIPÓTESE     → O que se pretende provar
03 SOLUÇÃO      → Como o problema foi abordado
04 EXPERIMENTO  → Como a solução foi testada
05 RESULTADO    → O que foi observado
06 LIMITAÇÕES   → O que ainda precisa ser resolvido
```

## Como Apresentar Cada Seção

### Problema
- Contextualização: situar no cenário real
- Impacto real: quantificar o prejuízo ou necessidade
- Relevância social/técnica: justificar por que merece atenção
- Exemplo correto: "Filas hospitalares geram desperdício de vagas" — não "vou fazer um app de hospital"

### Hipótese e Experimento
- Hipótese clara e testável
- Protocolo experimental definido
- Variáveis: independentes, dependentes e controle
- Grupo de controle e grupo de teste
- Baseline: referência de comparação **antes** da solução

> "Sem experimento controlado, não existe validação."

### Resultados
- Métricas: números objetivos que provam o efeito
- Antes / Depois: comparação direta com baseline
- Gráficos com eixos, legendas e unidades
- Exemplos de métricas adequadas:
  - ↓ 40% redução de tempo
  - ↑ 25% aumento de acurácia
  - ↓ 60% redução de erro

### Análise Crítica dos Resultados
- **Por que funcionou?** — explique o mecanismo técnico
- **Quando falha?** — identifique cenários onde não se aplica
- **Limitações** — seja honesto sobre as restrições
- **Melhorias** — proponha próximos passos e trabalhos futuros

> "Resultado sem interpretação não é pesquisa."

## Estrutura do Vídeo Final (5 minutos)
| Timestamp | Seção |
|-----------|-------|
| ~0:40 | Problema |
| ~1:10 | Objetivo |
| ~1:50 | Solução |
| ~2:40 | MVP |
| ~3:20 | Experimento |
| ~4:10 | Resultado |
| ~5:00 | Conclusão |

## Novo Padrão da Disciplina
| Requisito | Regra |
|-----------|-------|
| Não basta construir | Código funcional é apenas o início |
| É obrigatório validar | Evidência experimental é requisito |
| É obrigatório explicar | Comunicação científica é competência |
| Engenharia exige evidência | Sem dados, é opinião, não engenharia |

> **"Quem não consegue explicar o resultado, não domina o projeto."**
